# 【Code With SOLO】用 SOLO 搭建一个研究生自我帮助小助手：找文件、读文档、分析大代码项目

## 1. 摘要

我用 TRAE SOLO 搭建了一个本地运行的“自我帮助小助手”，解决研究生日常桌面文件混乱、临时找资料困难，以及面对大型代码项目不知道从哪里读起的问题。最终实现了一个可交互 Web 工具：输入文件名或关键词即可返回文件路径和内容摘要；输入代码项目路径后，可以先查看包/模块说明，再点击某个包查看函数级解释。

这个作品不只是生成脚本，而是把“文件查找、文档摘要、代码结构分析、可视化展示、本地服务”串成了一套完整工作流。

## 2. 背景

我是一名研究生，平时既要处理论文、组会记录、课程资料、PPT、实验材料，也要阅读和修改大型代码项目。时间一长，桌面和文件夹里会积累大量文档、代码、压缩包和图片，有时候我只记得文件名里的几个字，比如 `leetcode`、`组会`、`论文`，但很难快速定位文件到底在哪。

另外，在做代码相关任务时，我经常会遇到大型项目，例如 `F:\Desktop\xla`。这类项目文件很多、目录层级深，如果直接人工阅读，会花很多时间在“这个包是做什么的”“哪个函数值得先看”这些前置问题上。

所以我希望用 TRAE SOLO 做一个本地小助手：

- 文件方面：输入文件名或关键词，自动找到文件路径，并给出大致内容。
- 代码方面：输入一个大项目路径，自动分析包/模块结构，再展示每个包下的函数作用。
- 隐私方面：默认本地运行，不上传我的私人资料；需要时再切换模型增强分析。

## 3. 实践过程

### 3.1 任务拆解

我一开始只想做一个“代码仓库体检工具”，但在和 SOLO 讨论后发现，真实需求其实不止代码，还有我平时散落在桌面和文件夹里的文档资料。

于是我把任务拆成两个核心工作流：

1. 文件查找工作流
   - 输入文件名片段或关键词
   - 支持搜索范围留空，默认扫描桌面、文档和当前工具目录
   - 返回匹配文件路径
   - 提取文档大致内容
   - 生成 Markdown 报告和 JSON 索引

2. 代码分析工作流
   - 输入一个代码项目目录
   - 扫描代码文件
   - 识别包/模块
   - 分析每个包大概负责什么
   - 点击某个包后展示函数名、位置、行数和函数作用
   - 支持离线分析，也支持豆包、DeepSeek、ChatGPT、Gemini 等模型增强

### 3.2 使用 SOLO 的能力

这次实践中，SOLO 主要承担了这些工作：

- 帮我把模糊需求拆成清晰的产品流程。
- 生成 Python 文件扫描脚本。
- 生成代码函数级分析脚本。
- 生成本地 HTTP 服务，让网页按钮可以真正调用脚本。
- 设计前端交互页面。
- 生成可视化图片和报告。
- 根据测试结果不断修改交互逻辑。

最终项目不是一个单独脚本，而是一个可以在浏览器中使用的小工具。

### 3.3 关键 Prompt / 操作过程

一开始我给 SOLO 的需求大概是：

```text
我想做一个“自我帮助小助手”，可以扫描我的代码项目和个人文档目录。
要求本地运行，不上传文件；能生成 Markdown 报告；能帮我找资料、总结代码项目风险、列出最近修改文件和大文件。
请先帮我设计功能，再生成一个可运行的 Python 命令行工具。
```

后面我又逐步追加需求：

```text
我输入我要找的文件名，或者文件名里的几个字后，就进行查找。
找到后显示文件所在路径，并输出文件大致内容。
```

```text
代码分析不是找文件，而是我输入一个大项目后，先分析每一个包是干什么的。
当我选择某一个包后，再显示这个包里每一个函数是干什么的。
```

```text
代码分析模式可以选择模型，配置放在当前目录 .env 里，支持豆包、DeepSeek、ChatGPT、Gemini。
```

SOLO 根据这些需求逐步生成和修改了脚本、页面和本地服务。

### 3.4 中间踩过的坑

第一个坑是：最初页面只是静态 HTML，使用 `file://` 打开时，浏览器不能直接运行本地 Python 脚本。  
后来 SOLO 帮我改成 `local_server.py`，通过 `http://127.0.0.1:8765/` 打开页面，按钮点击后由本地服务调用 Python 脚本。

第二个坑是：文件查找一开始只是展示命令，用户体验不好。  
后来改成点击按钮后自动运行脚本，结果页直接展示文件路径和内容摘要，命令只作为记录放在后面。

第三个坑是：桌面下有大型代码仓库，默认扫描可能被大目录拖慢。  
后来 SOLO 调整了默认搜索范围，把桌面一级子文件夹也作为搜索根，提高了找到目标文件的概率。

第四个坑是：代码分析最初只给函数列表，但我真正需要的是先理解“包/模块”。  
后来改成先展示包/模块列表，点击某个包后再展示函数说明，更符合阅读大型项目的习惯。

## 5. 效果与总结

原来我临时找一个文件，可能需要在桌面和多个文件夹里反复翻，尤其只记得文件名几个字的时候非常低效。现在只要输入关键词，比如 `leetcode`，工具就能自动找到文件路径，并展示大致内容。

原来阅读一个大型代码项目时，我需要先手动看目录、猜模块职责、再逐个点文件。现在工具可以先生成包/模块地图，再点击某个包查看函数说明，代码阅读入口更清晰。

这次实践里，SOLO 不只是帮我写代码，还帮我完成了：

- 需求拆解
- 工具设计
- 后端服务实现
- 前端页面搭建
- 文件摘要提取
- 代码函数分析
- 模型配置设计
- 多轮交互优化

我觉得这套方法是可复用的：先从一个真实痛点出发，不急着做“大而全”的系统，而是让 SOLO 帮忙把工作流拆成可运行的小工具，再通过真实测试不断调整。对研究生来说，这种“个人工作流自动化”很实用，可以把大量重复的查找、整理、阅读入口构建工作交给 AI 辅助完成。

最终提效效果：

- 文件查找：从手动翻找十几分钟，变成输入关键词后自动返回路径和摘要。
- 代码阅读：从直接面对大项目目录，变成先看包/模块地图，再看函数说明。
- 报告整理：工具自动生成 Markdown 报告和可视化图片，方便后续复盘和分享。

这次作品让我最大的感受是：AI 编程不只是“帮我写一段代码”，更重要的是帮我把一个真实工作场景变成可复用的工具。
